## Completed This Session (Phase 2 — M13)

- M13.3: structured tool output. **Renderer**: new pure validator `src/renderer/src/components/results/output-schema-validator.ts` (`validateAgainstOutputSchema`) implements a SchemaForm-scoped JSON Schema subset — top-level object with `properties`, `required`, primitive `type` checks (`string` / `number` / `integer` / `boolean` / `null`), `enum`, `array` with `items`, and union types via `type: string[]`. Returns `{ ok: true } | { ok: false; errors: { path, message }[] }`. Deep `$ref` / `oneOf` / `anyOf` are intentionally out of scope per decision #14 (deferred to M20). **ResultRenderer** picked up an `outputSchema` prop and an extracted `structuredContent` helper; when both are present, the result viewer adds a third "Structured" toggle alongside Tree / Raw that surfaces (a) an emerald success banner when validation passes, (b) a rose-bordered banner listing path-pinned errors when it fails, and (c) a JsonNode view of `structuredContent`. Tree / Raw still render the full `CallToolResult`, so the raw `content` payload remains available even when validation fails (per spec contract that servers MUST always include `content`). **Discovery store**: `invokeTool` now snapshots the invoked tool's `outputSchema` into a new `activeOutputSchema` slot (cleared by `clearResult` / `loadResource` / `loadPrompt` so a tool's schema does not leak into a subsequent resource/prompt result). **Wiring**: `DiscoveryPanel` accepts `activeOutputSchema`, forwards to `ResultRenderer`, and keys the result component by `activeResultTitle` so each new invocation remounts the view with a fresh default mode. **Tests**: `tests/output-schema-validator.test.ts` (11 tests) covers non-object schemas, valid payloads, missing required fields, wrong primitive types, integer non-integer rejection, `enum` mismatch, non-object payloads, array item validation, union types, opaque extra keys, and multi-error accumulation. Full suite: 139 tests pass.

- M13.2: tool annotations + destructive-tool confirmation gate. **Shared IPC**: introduced typed `ToolAnnotations` (`title?`, `readOnlyHint?`, `destructiveHint?`, `idempotentHint?`, `openWorldHint?`) and `ToolIcon` (`src`, optional `mimeType`, `sizes`, `theme: 'light'|'dark'`) in `src/shared/ipc.ts`; `DiscoveryTool` now exposes `title?`, typed `annotations?: ToolAnnotations`, and `icons?: ToolIcon[]` instead of opaque `Record<string, unknown>`. **Main**: `src/main/mcp/session/discovery.ts` gained `projectToolAnnotations` and `projectToolIcons` helpers that defensively pick recognized fields off the SDK tool object, drop wrong-typed values, and omit the field entirely when no recognized data remains (spec is explicit that annotations are untrusted hints, so the projection treats them as such). **Renderer**: new `components/discovery/ToolBadges.tsx` renders colored badges for the four boolean hints with explanatory `title=` tooltips; new `components/discovery/DestructiveConfirmModal.tsx` is a Cancel / Invoke-anyway modal rendered when the user submits a tool with `destructiveHint: true`. `DiscoveryPanel` now shows `tool.title` (or `annotations.title`) as the display name above the monospace `tool.name`, mounts `ToolBadges` under the description, and routes destructive invocations through the confirmation gate before forwarding to `onInvokeTool`. **Tests**: new `tests/discovery-list-tools.test.ts` (7 tests) covers minimal projection, title + description pass-through, recognized boolean fields only, stripping of unknown / wrong-typed annotation values, defensive icon projection (rejects null entries, entries without `src`, non-string sizes, invalid themes), and omission of `icons` when no valid entry remains. Full suite: 128 tests pass.

- M13.1: spec/SDK reconciliation for M13 Server Feature Rendering. Anchored M13 to MCP spec 2025-11-25 + SDK 1.29.0 (same baseline as decision #13). Recorded findings in `context/DECISIONS.md` decision #14 covering five surface areas: (1) **tool annotations** — exact field list (`title`, `readOnlyHint`, `destructiveHint`, `idempotentHint`, `openWorldHint`), tool-level additions (`title`, `icons`), and the deferral of `execution.taskSupport` until P7's `tasks` capability lands; (2) **structured tool output** — `CallToolResult.structuredContent` + `outputSchema` validation with `content` always-fallback contract per spec; deep `$ref`/`oneOf` deferred to M20; (3) **completion** — `Client.complete` shape, `ref/prompt`+`ref/resource` union (note `ResourceReferenceSchema` deprecated in favor of `ResourceTemplateReferenceSchema`), new `params.context.arguments` for cascading completions, gated on `serverCapabilities.completions`; (4) **resource subscriptions** — `Client.subscribeResource`/`unsubscribeResource` + `notifications/resources/updated`, gated on `serverCapabilities.resources.subscribe`, session-scoped (no cross-reconnect persistence), drained on disconnect/shutdown/error like inflight/elicitation; (5) **server logging** — fixed eight-level `LoggingLevelSchema` (RFC 5424), `Client.setLoggingLevel`, `notifications/message` with `data: unknown`, no separate persistence (already captured by `MessageRecorder`); Logs sub-panel is a renderer projection over the protocol message stream filtered by `method === 'notifications/message'`.

## Completed This Session (Phase 2 — M12)

- M12.6: added in-flight operation tracking with progress + cancellation. Main: new `src/main/mcp/session/inflight.ts` exports `InflightOperationsStore` (Map keyed by `operationId`, methods `start` / `recordProgress` / `complete` / `cancel` / `rejectBySession` / `list`, `onChange` listener fan-out). Each entry carries an `AbortController` whose signal is forwarded to the SDK call via `RequestOptions.signal`; calling `cancel` aborts the controller, which causes the SDK to raise `AbortError` from the in-flight `request()` and auto-emit `notifications/cancelled` over the transport. `discovery.callTool/readResource/getPrompt` now accept `{ signal, onProgress }` and forward through `RequestOptions` (`signal`, `onprogress`). `SessionManager.callTool/readResource/getPrompt` wrap each invocation in a `runTracked` helper that allocates an `operationId`, registers the entry, threads signal + progress callback into the discovery call, and removes the entry in `finally`. New public methods `listInflightOperations`, `cancelInflightOperation`, `onInflightChange`. Inflight entries are drained on disconnect / shutdown / error (aborting any still-pending controller). IPC: added 3 channels (`mcp-inflight:list`, `mcp-inflight:cancel`, `mcp-inflight:stream`), matching Zod schemas with compile-time `Equals` guards, push-stream subscription in `src/main/index.ts` that broadcasts the full operation list on store change, preload bridge entries (`listInflightOperations` / `cancelInflightOperation` / `subscribeInflightOperations`), and `ipc-fallback` noops. Renderer: new `useInflightStore` (Zustand) seeds via `listInflightOperations()` and subscribes; `InflightOperationsPanel` mounts above the Discovery panel and only renders when there is at least one in-flight op — each card shows kind, label, start time, progress text + bar (when `total` is known), and a Cancel button. Added `INFLIGHT_OPERATION_NOT_FOUND` to `AppErrorCode`. Tests: `tests/session-inflight.test.ts` (9 tests) covers store start/recordProgress/complete/cancel semantics, AbortController abort assertion, duplicate id rejection, session-scoped drain, and summary projection.
